Here is the complete Functional Requirements and UI/UX Blueprint for the Admin Portal (Content Management System).
To bring your highly structured psychological "Mind Tree" to life, your writers, psychologists, and editors cannot be writing raw code. They need a powerful, desktop-web-based "Visual Object Builder" (heavily inspired by tools like Notion, Webflow, or Miro).
This portal translates your complex database hierarchy into a simple "fill-in-the-blanks" interface with robust Copy, Delete, Add, and Comment tools.
🛠️ THE ADMIN PORTAL: The "Mind Object" Builder (CMS)
Target Platform: Secure Desktop Web Application (Optimized for mouse navigation, typing, and drag-and-drop file uploads).
Target Users: Admin Users, Content Writers, Psychologists, Audio Editors, and Reviewers.
1. Global UI Layout (The 3-Panel Workspace)
When an admin clicks [ + Create New Book Breakdown ], the screen divides into an intuitive, high-productivity workspace:
 * Left Panel (The "Mind" Tree): A visual, sticky navigation menu showing the exact JSON structure of the book (🎯 Thesis ➡️ 📑 Chapters ➡️ 📜 Principles, etc.). Admins can click any section to jump right to it or drag-and-drop to reorder the flow.
 * Center Panel (The Block Builder): The main "Fill-in-the-Blanks" workspace where admins type text, upload pictures ("pics"), and attach audio.
 * Right Panel (Live Mobile Emulator): A digital iPhone/Android frame. As the admin types text or uploads a picture in the center panel, the phone screen updates in real-time so they can see exactly how the "Tap-Designed" UI will look to the end-user.
2. Universal "Block Controls" (Copy, Delete, Add, Comment)
To meet your exact requirement, the builder is purely object-oriented. Whenever an admin hovers their mouse over any element in the center panel (a Chapter screen, a Principle, an Exercise), a floating action toolbar appears attached specifically to that object:
 * ➕ Add New: Inserts a blank template of that exact format directly below it.
 * 📄 Copy (Duplicate): Instantly clones the object (including its text, uploaded pics, and audio). Crucial time-saver: A writer copies a Chapter Slide so they don't have to re-upload the background picture, they just change the text for the next tap.
 * 🗑️ Delete: Removes the object. (Safety Failsafe: If an admin clicks delete on a "Principle" that has a nested "Story" attached to it, a red warning modal pops up: "Deleting this Principle will also delete its attached Story. Are you sure?").
 * 💬 Comment (Review Mode): Opens a chat thread attached specifically to that block.
   * Workflow: A Senior Editor highlights a junior writer's "Story" block, clicks Comment, and types: "@Sarah, this text is too long for a mobile screen. Please make it punchier." The block turns yellow until Sarah resolves the comment.
3. Module-by-Module UI (The "Fill-in-the-Blanks" Editor)
Here is exactly how the center panel forms are designed for your specific psychological taxonomy:
A. Global Setup & 🎯 Core Thesis
 * Fields: Book Title & Author Name (Standard text fields).
 * Master Media: Two large drag-and-drop zones. One for [ Drop Book Cover Pic Here ] and one for [ Drop Master Audio Summary (MP3) Here ].
 * Core Thesis: A bold text box locked to a strict 200-character limit. The UI physically stops the writer from typing more, forcing them to distill the ultimate "Big Idea" concisely.
B. 📑 Tap-Designed Chapter Summaries Builder
 * UI Layout: Looks like an Instagram Story or Canva slideshow timeline.
 * Inputs: Admins click [ + Add Chapter ]. Inside that chapter timeline, they click [ + Add Tap-Screen ].
   * For every "Screen" object, they fill in:
     * Background Upload: Drop a picture or select a hex color code.
     * Text Box: Type the 1-2 sentence punchy summary of the chapter event.
     * Audio Snippet (Optional): Attach the 10-second voiceover file for this specific screen.
C. 🧠 Mental Models Builder (Interactive Visuals)
 * UI Layout: Built specifically for the "Tap-to-Reveal" mobile animation feature.
 * Inputs:
   * Base Pic Upload: Admin drops the foundational visual (e.g., a blank pyramid or blank flowchart).
   * Reveal Steps: Admin clicks [ + Add Tap Step ].
   * For Step 1, they upload the picture of the first layer filling in, and add the text explanation. They repeat this for Step 2, Step 3, etc. The right-panel emulator lets them click the phone screen to test the animation.
D. 📜 Principles & Stories Builder (Parent/Child Linker)
 * UI Layout: A visually nested block to enforce your rule that every principle must have proof.
 * Inputs:
   * Parent (The Rule): Admin clicks [ + Add Principle ]. A large text field appears.
   * Child (The Proof): Directly inside that box, indented to the right, is a mandatory nested block: [ ↳ Attach Story ]. The admin types the real-world anecdote here and can drop an optional picture (e.g., a photo of the historical figure the story is about).
E. 🚫 Common Mistakes (Anti-Patterns Builder)
 * UI Layout: A clean, side-by-side split-screen input form.
 * Inputs:
   * Left Side (Red Box): Labeled "The Mistake (What people do wrong)". Admin types the flawed behavior.
   * Right Side (Green Box): Labeled "The Fix (The psychological correction)".
   * (The CMS automatically maps this into the sleek "Do This / Not That" comparison card for the mobile app).
F. ✍️ Exercises Builder (Impact Ranked)
 * UI Layout: A dynamic form builder (similar to Google Forms).
 * Inputs:
   * Prompt Text Box: (e.g., "What is your biggest trigger today?").
   * Input Type Selector: Admin chooses how the mobile user will answer: [ Text Journal, Multiple Choice, or 1-10 Slider ].
   * 🔥 Impact Ranking Dropdown (Mandatory): Admin must tag the exercise as: [ High Impact ], [ Medium Impact ], or [ Low Impact ]. (The backend database uses this tag to automatically force the High Impact exercises to the top of the user's mobile screen).
G. ✅ Action Items Builder
 * UI Layout: A rapid-fire list builder.
 * Inputs:
   * Admin types the task.
   * Toggle Switch: Admin flips a physical UI switch to tag the object as either [ ⚡ Immediate (Do Today) ] or [ 🗓️ Long-Term (Habit) ].
4. The Publishing Engine (How it goes live)
 * F-1. Draft & Auto-Save: Building these highly curated objects takes time. The portal auto-saves every 3 seconds to the cloud. Admins can leave a book in "Draft" status while waiting for the voice actors to finish the audio files.
 * F-2. The Compile & Publish Logic: When the book is fully approved and all comments are resolved, the Master Admin clicks a giant green [ 🚀 Publish to App ] button.
 * F-3. The Developer Magic: The CMS backend instantly takes all these individual UI blocks, form inputs, pictures, and audio files, compiles them into the exact strict hierarchical JSON tree you designed, and pushes it to the cloud database (AWS/Google Cloud). The new interactive summary instantly appears on users' iOS and Android phones without requiring an app update.
