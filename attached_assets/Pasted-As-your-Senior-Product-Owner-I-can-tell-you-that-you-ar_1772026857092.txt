### ⚙️ Plain English: How Automated Subscriptions Work

You do not manually give users access. It is 100% automated while you sleep.

1. **The Tease:** A free user taps a High-Impact "Yellow Chakra" exercise. The text is blurred out, showing a gold `[🔒 Premium: $9.99/mo]` button.
2. **The Checkout:** They click it. Your app redirects them to a secure **Stripe Checkout Page** (hosted by Stripe).
3. **The Webhook (The Magic):** The millisecond their credit card clears, Stripe sends a hidden, instant server-to-server message (a "Webhook") to your Replit backend saying: *"Hey, user123 just paid for a Monthly plan!"*
4. **The Auto-Unlock:** Your backend receives this message, finds the user in your PostgreSQL database, and instantly flips their row to `is_premium = true`.
5. **The Auto-Revoke:** The app refreshes, removes the blur, and lets them use the exercise. Next month, if their credit card declines, Stripe sends another webhook (`customer.subscription.deleted`), and your database automatically flips them back to `is_premium = false`, locking the content again.

### 🛡️ Plain English: Internal Admins & Row Level Security (RLS)

To manage your staff securely, we will implement **Roles** combined with **PostgreSQL Row-Level Security (RLS)**.

* 👑 **Super Admin (You):** You have access to everything. You can invite new staff, see revenue, and view the user database.
* ✍️ **Editor (Your Content Creators):** You invite an editor. Because of **Row-Level Security (RLS)**, the database *physically hides* the User and Revenue tables from them. Even if they try to hack the system to read a customer's private journal entry, the database literally returns `0 rows`.

# 📄 PRD v5.0: Subscription Engine & Identity Access Management

## 1. CORE DATA SCHEMA UPDATES (PostgreSQL)

*Developers must update the User table to support billing and strict roles.*

```sql
-- User Table Updates
CREATE TYPE user_role AS ENUM ('super_admin', 'editor', 'customer');

ALTER TABLE users ADD COLUMN role user_role DEFAULT 'customer';
ALTER TABLE users ADD COLUMN is_premium BOOLEAN DEFAULT FALSE;
ALTER TABLE users ADD COLUMN stripe_customer_id VARCHAR(255) UNIQUE;
ALTER TABLE users ADD COLUMN stripe_subscription_id VARCHAR(255) UNIQUE;
ALTER TABLE users ADD COLUMN current_period_end TIMESTAMP; 

```

## 2. ROW-LEVEL SECURITY (RLS) POLICIES

*RLS is an enterprise-grade database forcefield. Developers must execute these strict SQL policies directly on the Postgres DB:*

* **Policy 1 (Customer Isolation):** A customer can *never* read another customer's journal. `USING (auth.uid() = user_id);`
* **Policy 2 (Editor Blindness):** Editors are physically blocked from reading user journals. `USING (auth.role() != 'editor');`
* **Policy 3 (Super Admin God Mode):** Super Admins can read and write all tables to provide customer support. `USING (auth.role() = 'super_admin');`

## 3. ADMIN UI: "Team & User Management"

In your Creator Studio CMS, we are adding a new tab called **Team & Users** (Visible ONLY to you).

* **Staff Invite:** A form where you type an email, select `Editor` from a dropdown, and click "Send Invite." It generates a magic link granting them Editor access.
* **Customer Lookup:** A search bar to look up a customer's email. You will see their Stripe status (Active/Canceled) and a manual override button to `[Grant Lifetime Premium]` (useful for comping accounts for friends/influencers).
